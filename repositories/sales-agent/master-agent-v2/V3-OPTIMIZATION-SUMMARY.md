### 3. Fix Baserow Datetime Validation (commit 43d1455)
**Files modified**: `SYSTEM-PROMPT-V3-OPTIMIZED.md` (lines 166-175, 274-276, 400-402)

**Problem**: LLM converted ISO 8601 timestamps to epoch milliseconds

**Error**:
```
field_7067: "Datetime has wrong format. Use YYYY-MM-DDThh:mm[:ss[.uuuuuu]][+HH:MM|-HH:MM|Z]"
```

**Example**:
```json
// Input (correct):
meta.now_ts: "2025-11-07T14:15:40.865Z"

// Output (incorrect):
addressee_ask_ts: 1699355740865  // ❌ epoch number

// Fixed output:
addressee_ask_ts: "2025-11-07T14:15:40.865Z"  // ✅ ISO string
```

**Solution**: Added explicit examples showing correct vs incorrect format throughout prompt

### 4. Fix Cooldown Detection (commit 43d1455)
**Files modified**: `SYSTEM-PROMPT-V3-OPTIMIZED.md` (lines 137-164)

**Problem**: Cooldown timestamps remained `null` when agent asked for name/email

**Root cause**: Initial greeting wasn't generated by Master Agent v3.0, so cooldown wasn't set

**Solution**: Implemented **retroactive detection** using regex on conversation history

**Logic**:
```javascript
// Check last assistant message in history
const lastAssistantMsg = history.filter(m => m.role === "assistant").slice(-1)[0];

// Retroactive detection for addressee_ask_ts
if (cooldowns.addressee_ask_ts === null && lastAssistantMsg) {
  const askedForName = /nombre|cómo te llam|quién sos|tu nombre/i.test(lastAssistantMsg.text);
  if (askedForName) {
    cooldowns.addressee_ask_ts = lastAssistantMsg.ts; // Use that message's timestamp
  }
}
```

**Result**: ✅ addressee_ask_ts correctly set, Baserow persisted successfully

### 6. Enforce Tool Calling (commit d8f2234)
**Files modified**: `SYSTEM-PROMPT-V3-OPTIMIZED.md` (lines 229-240, 244-270)

**Problem**: Agent said "te envío la propuesta" but `has_tool_calls: false`

**User feedback**: "no funciona bien"

**Example of bug**:
```json
{
  "message": {
    "text": "Te la envío ahora mismo al email..."  // ❌ Says it's sending
  }
  // ❌ No tool_calls array present!
}
```

**Solution**: Multiple enforcement layers

**Layer 1: Mutual Exclusion Rule** (lines 224-235):
```
IF missing data:
  → ASK for it (message only, NO tool_calls)
  → STOP HERE

IF all data present:
  → YOU MUST CALL THE TOOL
  → Include tool_calls in your JSON output
```

**Layer 2: Anti-Hallucination Warnings** (lines 237-240):
- If you say "te envío", "te mando", "te llega" → YOU MUST include `tool_calls`
- If you DON'T include `tool_calls` → DON'T say you're sending anything
- The tool will actually send the email, so you MUST call it

**Layer 3: Complete JSON Example** (lines 244-270):
Shows exact structure of tool_calls array inside message object

**Layer 4: Emphasized Placement** (line 270):
"CRITICAL: The tool_calls array goes INSIDE message, NOT at root level!"

## Key Learnings

### 1. Prompt Engineering
- **Less is more**: 66% reduction improved LLM comprehension
- **Examples over explanations**: Show JSON structures, don't just describe them
- **Explicit > Implicit**: LLM needs very explicit instructions (e.g., "NEVER ASK")

### 2. Tool Calling
- **Multiple enforcement layers needed**: One instruction wasn't enough
- **Show exact structure**: Provide complete JSON example with tool_calls array
- **Placement matters**: Emphasize WHERE tool_calls goes (inside message)

### 3. Data Format Validation
- **LLM will transform formats**: ISO 8601 → epoch, strings → numbers
- **Show correct vs incorrect**: Visual examples prevent conversion
- **Copy exact values**: Instruct to copy meta.now_ts EXACTLY

### 4. Testing Strategy
- **Test incrementally**: Each fix tested before moving to next
- **Document expected output**: Clear success criteria for each test
- **Monitor logs**: Output Main logs critical for debugging
